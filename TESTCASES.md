# TESTCASES - Avito QA Trainee Assignment

Выбранный вариант: **Задание 2.1 (API)**.

Базовый URL: `https://qa-internship.avito.com`

## 1) Позитивные сценарии

### TC-API-001 - Создать объявление с валидными данными
- **Предусловия:** API доступен.
- **Шаги:**
  1. Отправить `POST /api/1/item` с валидным body (`sellerID`, `name`, `price`, `statistics`).
- **Ожидаемый результат:**
  - `200 OK`
  - В ответе `POST` возвращается поле `status` со строкой вида `Сохранили объявление - <uuid>`
  - UUID из `status` используется для последующей валидации данных через `GET /api/1/item/{id}`.

### TC-API-002 - Получить объявление по id
- **Предусловия:** есть созданное объявление.
- **Шаги:**
  1. Создать объявление.
  2. Запросить `GET /api/1/item/{id}`.
- **Ожидаемый результат:**
  - `200 OK`
  - В ответе есть объявление с нужным `id` и корректными полями.

### TC-API-003 - Получить все объявления продавца
- **Предусловия:** есть минимум одно объявление с `sellerID = X`.
- **Шаги:**
  1. Создать объявление c `sellerID = X`.
  2. Выполнить `GET /api/1/{sellerID}/item`.
- **Ожидаемый результат:**
  - `200 OK`
  - Вернулся массив объявлений
  - В массиве есть созданное объявление
  - У каждого элемента `sellerId = X`.

### TC-API-004 - Получить статистику по item id
- **Предусловия:** есть созданное объявление.
- **Шаги:**
  1. Создать объявление с заданной `statistics`.
  2. Выполнить `GET /api/1/statistic/{id}`.
- **Ожидаемый результат:**
  - `200 OK`
  - Вернулся массив статистики
  - Присутствуют поля `likes`, `viewCount`, `contacts`
  - Значения соответствуют созданному объявлению.

## 2) Негативные сценарии

### TC-API-005 - Создание объявления без обязательного поля
- **Шаги:**
  1. Отправить `POST /api/1/item` без поля `name`.
- **Ожидаемый результат:**
  - `400 Bad Request`
  - Сообщение об ошибке валидации.

### TC-API-006 - Получить объявление по несуществующему id
- **Шаги:**
  1. Выполнить `GET /api/1/item/{random_uuid}` (id не существует).
- **Ожидаемый результат:**
  - `404 Not Found`.

### TC-API-007 - Получить объявление по некорректному id
- **Шаги:**
  1. Выполнить `GET /api/1/item/invalid-id`.
- **Ожидаемый результат:**
  - `400 Bad Request`.

## 3) Corner / E2E сценарии

### TC-API-008 - Проверка идемпотентности POST
- **Шаги:**
  1. Дважды отправить один и тот же payload в `POST /api/1/item`.
- **Ожидаемый результат:**
  - Оба запроса успешны
  - Созданы разные объявления с разными `id` (POST не идемпотентен).

### TC-API-009 - Сквозной сценарий create -> get by id -> get by seller -> get stats
- **Шаги:**
  1. Создать объявление.
  2. Получить по id.
  3. Получить список объявлений продавца.
  4. Получить статистику.
- **Ожидаемый результат:**
  - На каждом шаге `200`
  - Все ответы консистентны по `id`/`sellerId`/`statistics`.

## 4) Нефункциональные проверки

### TC-API-010 - Время ответа ключевых ручек
- **Шаги:**
  1. Измерить время `POST /api/1/item` и `GET /api/1/item/{id}`.
- **Ожидаемый результат:**
  - Время ответа каждой ручки < 2 сек при одиночном запросе.

### TC-API-011 - Независимость и воспроизводимость тестов
- **Шаги:**
  1. Запустить весь набор тестов несколько раз.
- **Ожидаемый результат:**
  - Тесты не зависят от порядка выполнения
  - Не используют фиксированные данные
  - Создают уникальные сущности (`sellerID` в диапазоне 111111–999999, уникальный `name`).

## 5) Контрактные проверки (JSON Schema)

### TC-API-012 - Контракт `POST /api/1/item`
- **Проверки:**
  - `200 OK`
  - Ответ соответствует JSON Schema `{"status": "<string with uuid>"}`.

### TC-API-013 - Контракт `GET /api/1/item/{id}`
- **Проверки:**
  - `200 OK`
  - Массив объявлений соответствует JSON Schema item-объекта (`id`, `sellerId`, `name`, `price`, `statistics`, `createdAt`).

### TC-API-014 - Контракт `GET /api/1/{sellerID}/item`
- **Проверки:**
  - `200 OK`
  - Массив соответствует JSON Schema item-объекта
  - `sellerId` всех элементов совпадает с запрошенным sellerID.

### TC-API-015 - Контракт `GET /api/1/statistic/{id}`
- **Проверки:**
  - `200 OK`
  - Массив статистики соответствует JSON Schema (`likes`, `viewCount`, `contacts`).

### TC-API-016 - Контракт `GET /api/2/statistic/{id}`
- **Проверки:** аналогично TC-API-015, но для v2.

### TC-API-017 - Контракт ошибок (`400/404`) для v1/v2
- **Проверки:**
  - Для невалидных и несуществующих идентификаторов тело ошибки соответствует JSON Schema:
    - `status: string`
    - `result.message: string`
    - `result.messages: object|null`.

## 6) Дополнительные ручки v2

### TC-API-018 - `DELETE /api/2/item/{id}` успешное удаление
- **Шаги:**
  1. Создать объявление через v1.
  2. Удалить через `DELETE /api/2/item/{id}`.
  3. Проверить, что `GET /api/1/item/{id}` возвращает ошибку.
- **Ожидаемый результат:**
  - `DELETE` возвращает `200`.
  - После удаления элемент недоступен (ошибка по `GET`).

### TC-API-019 - `DELETE /api/2/item/{id}` с несуществующим id
- **Ожидаемый результат:**
  - Возвращается ошибка (`404`) с валидной структурой error-response.

## 7) Полная матрица валидаций payload

### TC-API-020 - Матрица полей `sellerID`, `name`, `price`, `statistics.*`
- **Проверки (parameterized):**
  - missing/null/empty string/string type/large values для `sellerID`, `name`, `price`
  - missing/null/string для `statistics.likes`, `statistics.viewCount`, `statistics.contacts`
  - missing/null для объекта `statistics`
- **Ожидаемый результат:**
  - Для валидных/допустимых комбинаций `200`
  - Для невалидных `400`
  - Для ошибок дополнительно валидируется структура и содержание error-response.
